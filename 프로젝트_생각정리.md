# Industry Insight Agent 생각 정리

## 1. 이 프로젝트에서 내가 중점적으로 생각한 것

### 1-1. 단순 분석 프로젝트가 아니라 자동화 시스템으로 보이게 만들기

이 프로젝트를 만들 때 가장 중요하게 본 것은 "뉴스를 분석했다"가 아니라
"사람이 하던 반복 업무를 자동화 가능한 시스템으로 바꿨다"는 점이다.

그래서 아래 흐름이 끊기지 않도록 생각했다.

`수집 -> 정제 -> 분류 -> 요약 -> 저장 -> 분석 -> 시각화`

핵심은 각각의 기능이 따로 있는 것이 아니라,
업무 흐름 단위로 연결되어 있어야 한다는 점이다.

### 1-2. LLM을 기능이 아니라 파이프라인 안의 처리 엔진으로 보기

이 프로젝트에서 LLM은 그냥 "한 번 호출해보는 도구"가 아니라,
비정형 텍스트를 구조화 데이터로 바꾸는 핵심 처리 엔진으로 사용했다.

즉, LLM을 쓴 것 자체보다 아래가 더 중요하다고 생각했다.

- 어떤 입력을 넣을지
- 어떤 카테고리 체계를 쓸지
- 어떤 형식으로 결과를 저장할지
- 후속 분석에 어떻게 연결할지

### 1-3. 실제 업무에서 쓸 수 있는 출력 만들기

뉴스 원문만 가지고 있으면 다시 사람이 읽어야 한다.
그래서 결과는 아래처럼 바로 활용 가능한 형태가 되도록 설계했다.

- `industry`
- `category`
- `summary`
- `source_name`
- `collection_method`
- `processed_at`

즉, "텍스트를 읽었다"가 아니라 "다음 단계에서 바로 쓸 수 있는 데이터로 바꿨다"가 중요하다.

### 1-4. 데모가 끊기지 않는 구조 만들기

포트폴리오 프로젝트는 발표나 면접에서 직접 보여줄 가능성이 높다.
그래서 네트워크, API 키, 외부 환경 문제 때문에 전체 흐름이 깨지지 않도록
fallback 구조를 넣는 것을 중요하게 생각했다.

- live RSS 수집이 안 되면 local seed fallback
- OpenAI API가 없으면 rule-based fallback

이 부분은 단순 편의 기능이 아니라,
"운영을 생각한 설계"라는 점에서 중요하다.

### 1-5. 햄프킹과 연결되는 방향으로 프로젝트를 재해석하기

이 프로젝트를 단순 뉴스 분석으로 설명하면 강점이 약하다.
그래서 아래 키워드로 보이도록 만드는 데 집중했다.

- 비정형 데이터 처리
- LLM 기반 자동화
- 업무 프로세스 구조화
- 향후 RPA나 Agent 확장 가능한 구조

즉, 결과보다도 "어떤 방식으로 문제를 구조화했는가"를 중요하게 봤다.

## 2. 만약 면접을 본다면 주요하게 생각하고 해야 할 것

### 2-1. 프로젝트를 소개할 때 첫 문장을 잘 잡기

면접에서는 프로젝트를 "뉴스 분석 프로젝트"라고 시작하지 않는 것이 좋다.
아래처럼 말하는 편이 더 강하다.

> 비정형 산업 뉴스 데이터를 자동 수집하고, LLM으로 분류와 요약을 수행한 뒤, 구조화 데이터와 대시보드로 연결하는 자동화 시스템을 구현했습니다.

이 문장 안에는 아래 요소가 모두 들어간다.

- 비정형 데이터
- 자동화
- LLM 활용
- 시스템 구현
- 결과 활용

### 2-2. 기술보다 문제와 업무 흐름을 먼저 설명하기

면접에서는 보통 "무엇을 만들었냐"보다 "왜 만들었고 어떤 문제를 해결했냐"가 더 중요하다.

좋은 흐름은 아래와 같다.

1. 뉴스 모니터링은 사람이 수작업으로 하기엔 양이 많다
2. 분류와 요약이 반복 업무가 된다
3. 비정형 데이터라 자동화가 어렵다
4. 그래서 수집부터 요약, 저장, 분석까지 이어지는 파이프라인으로 설계했다

즉, 기능 설명 전에 업무 문제를 먼저 말해야 한다.

### 2-3. LLM을 쓴 이유를 분명하게 말하기

면접에서는 "왜 그냥 규칙 기반으로 안 했나요?" 같은 질문이 나올 수 있다.
그때는 아래처럼 답하면 좋다.

- 뉴스는 비정형 텍스트라 규칙만으로 일반화가 어렵다
- 의미 단위 분류와 요약은 LLM이 더 유연하다
- 다만 안정성을 위해 fallback도 같이 설계했다

즉, LLM을 맹신한 것이 아니라
유연성과 안정성의 균형을 잡으려 했다고 말하는 것이 좋다.

### 2-4. 구현보다 설계 의도를 말해야 한다

면접에서는 "이 코드를 작성했습니다"보다 아래를 말하는 것이 더 강하다.

- 왜 수집 단계와 분석 단계를 분리했는지
- 왜 결과를 CSV로 저장했는지
- 왜 category와 summary를 분리했는지
- 왜 dashboard까지 붙였는지
- 왜 fallback을 넣었는지

즉, "무엇을 했다"보다 "왜 그렇게 설계했는가"를 준비해야 한다.

### 2-5. 한계도 솔직하게 말할 준비를 하기

오히려 한계를 명확히 아는 것이 더 좋다.

예를 들면 아래처럼 말할 수 있다.

- 현재는 결과 평가 지표가 아직 약하다
- 실시간 스트리밍까지는 구현하지 못했다
- live RSS는 환경에 따라 제약이 있을 수 있다
- 현재 taxonomy는 더 정교하게 다듬을 수 있다

중요한 것은 한계를 말한 뒤 개선 방향까지 같이 말하는 것이다.

### 2-6. 햄프킹 맞춤으로 반드시 연결하기

면접에서는 이 프로젝트가 왜 햄프킹과 맞는지를 스스로 연결해서 말해야 한다.

핵심 연결 포인트는 아래다.

- 비정형 텍스트 데이터를 처리했다
- 사람이 하던 반복 분류/요약 업무를 자동화했다
- 수집부터 결과 활용까지 이어지는 파이프라인을 설계했다
- 이후 Agent, 알림, RPA로 확장 가능한 구조다

즉, "이 프로젝트를 통해 AI 기능을 업무 자동화 구조 안에 넣는 방식을 고민했다"라고 말하면 좋다.

## 3. 면접에서 실제로 특히 조심해야 할 것

### 3-1. "LLM 써봤다" 수준으로 말하지 않기

그냥 API 호출 정도로 들리면 프로젝트 깊이가 약해 보인다.
반드시 아래 단어들을 같이 써야 한다.

- workflow
- pipeline
- structured output
- automation
- fallback
- decision support

### 3-2. 너무 기능 나열식으로 말하지 않기

아래처럼 말하면 약하다.

- 수집했습니다
- 분류했습니다
- 요약했습니다
- 시각화했습니다

대신 아래처럼 묶어서 말하는 것이 좋다.

- 입력된 비정형 데이터를 분석 가능한 구조로 전환했다
- 반복적인 뉴스 모니터링 업무를 자동화 흐름으로 설계했다
- AI 결과를 후속 분석과 시각화까지 연결했다

### 3-3. 과장하지 않기

실제로 구현하지 않은 부분까지 "다 됩니다"라고 하면 오히려 위험하다.

예를 들면 아래는 구분해서 말해야 한다.

- 구현한 것: 수집 fallback 구조, 분류/요약, 저장, 대시보드
- 확장 예정: 실시간 스트리밍, 알림, RPA 연동, 정량 평가 고도화

## 4. 이 프로젝트 안에서 꼭 공부해야 할 내용

### 4-1. LLM 기본기

반드시 이해해야 할 것:

- 프롬프트 설계
- temperature가 결과에 미치는 영향
- hallucination 가능성
- 토큰 비용 개념
- 분류와 요약처럼 태스크별 프롬프트 차이
- structured output이 왜 중요한지

이 프로젝트에서는 특히 "LLM을 어디까지 믿고 어디서 보완할 것인가"를 이해해야 한다.

### 4-2. OpenAI API 사용 방식

꼭 공부할 것:

- API 호출 구조
- 모델 선택 기준
- 오류 처리 방법
- 응답 실패 시 fallback 전략
- rate limit이나 비용 이슈를 어떻게 고려할지

면접에서는 "실무에서 API를 어떻게 안정적으로 쓸 것인가" 질문이 나올 수 있다.

### 4-3. 데이터 파이프라인 사고

이 프로젝트를 잘 설명하려면 ETL 관점을 알아야 한다.

- Extract: 뉴스 수집
- Transform: 정제, 정규화, 분류, 요약
- Load: CSV 저장, 대시보드 연결

즉, 단순 pandas 처리보다 파이프라인 관점으로 공부해야 한다.

### 4-4. 비정형 데이터 전처리

꼭 이해해야 할 것:

- 텍스트 정제
- 결측치 처리
- 컬럼 매핑
- 중복 제거
- source schema가 다를 때 어떻게 맞출지

뉴스 데이터는 원천마다 형식이 달라서 이 부분이 굉장히 중요하다.

### 4-5. 분류 체계 설계

카테고리는 그냥 정하는 것이 아니라,
업무적으로 의미가 있어야 한다.

공부할 것:

- taxonomy 설계 방식
- 카테고리가 너무 많을 때와 너무 적을 때의 문제
- 분류 기준을 어떻게 문서화할지
- 사람 기준과 모델 기준을 어떻게 맞출지

### 4-6. 요약 품질 평가

요약은 "그럴듯하다"로 끝내면 안 된다.
아래를 생각해야 한다.

- 핵심 사실이 들어갔는지
- 불필요한 내용이 줄었는지
- 왜 중요한지가 드러나는지
- 사람이 검토하기 쉬운 길이인지

즉, 요약도 품질 기준이 필요하다.

### 4-7. 분석과 시각화

지금 프로젝트에는 아래 분석이 들어간다.

- 카테고리 분포
- 산업별 비교
- 키워드 분석

여기서 더 공부하면 좋은 것:

- 시계열 추세 분석
- 이상치 감지
- 중요 이벤트 탐지
- 감성 분석

### 4-8. 자동화 운영 관점

햄프킹과 가장 연결되는 부분이라 꼭 공부하는 것이 좋다.

- 스케줄링
- 재시도
- 실패 로그 관리
- 알림
- 배치 실행
- 후속 시스템 연동

즉, "모델이 잘 동작하는가"를 넘어서
"운영 가능한 자동화인가"를 생각해야 한다.

## 6. 면접 전 꼭 준비할 답변

아래 질문은 반드시 준비해두는 것이 좋다.

### 왜 이 프로젝트를 했나요

비정형 뉴스 데이터를 사람이 반복적으로 읽고 정리하는 업무를 자동화 가능한 형태로 바꾸고 싶었기 때문입니다.

### 이 프로젝트의 핵심은 무엇인가요

LLM을 사용한 것 자체보다, 수집부터 분석, 저장, 시각화까지 이어지는 자동화 파이프라인으로 만든 것이 핵심입니다.

### 햄프킹과 어떤 관련이 있나요

비정형 데이터 처리, 반복 업무 자동화, LLM 기능의 시스템화라는 점에서 햄프킹의 방향성과 맞닿아 있습니다.

### 가장 중요하게 설계한 부분은 무엇인가요

데모가 끊기지 않도록 fallback을 넣고, 결과를 구조화 데이터로 저장해 후속 분석과 확장이 가능하도록 만든 부분입니다.

### 보완하고 싶은 부분은 무엇인가요

실제 LLM 결과 평가, 실시간 수집 안정화, 알림 및 RPA 연동까지 확장하고 싶습니다.

## 7. 최종 정리

이 프로젝트에서 가장 중요한 포인트는 아래 세 줄로 정리할 수 있다.

1. 비정형 뉴스 데이터를 구조화 데이터로 바꿨다
2. LLM을 단순 호출이 아니라 자동화 파이프라인 안에 넣었다
3. 사람이 하던 반복 업무를 시스템 형태로 설계했다

면접에서는 이 프로젝트를 "뉴스 분석"이 아니라
"업무 자동화 시스템 설계 경험"으로 설명하는 것이 가장 강하다.
